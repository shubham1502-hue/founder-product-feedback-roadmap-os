# Founder Use Case

This example uses the synthetic sample dataset included in the repo.

A seed-stage founder has 35 feedback items from sales calls, onboarding notes, support tickets, renewal reviews, expansion conversations, and founder demo observations. The team has a long list of requests, but no shared answer to what should be built now.

The founder runs:

```bash
make run
```

The first file to open is:

```text
outputs/founder_roadmap_memo.md
```

The founder can quickly see:

- Which roadmap items are build-now candidates
- Which items need customer validation before engineering scope
- Which product gaps are blocking revenue
- Which gaps create retention risk
- Which asks may unlock expansion
- Which loud requests should be avoided
- Which issues should be solved through onboarding, sales narrative, support, documentation, or CS instead of product work

In the sample data, repeated integration requests, reporting gaps, enterprise controls, activation bugs, and team permissions rise above one-off customization requests. This gives the founder a practical review agenda for product, CS, sales, support, and GTM owners.

The value is not the score by itself. The value is that every score is connected to owner action.
